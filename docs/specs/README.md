# Vibe学习记 Spec 分配矩阵

本文把最终产品拆成 Phase 1 和 Phase 2 各 8 个学生任务。它先确定任务边界、并行关系和验收入口；每一项的完整用户故事、交互细节和 Given / When / Then 场景随后分别写入 `phase-1/` 与 `phase-2/`。

产品行为以 [Product Design](../product-design.md) 为准，视觉实现以 [UI Design](../ui-foundation-design.md) 为准，公共工程能力以 [Starter Kit Contract](../starter-kit-contract.md) 为准。

## 1. 分配原则

- 每个 Spec 必须交付一个可以在开发者工具中看见、操作和验收的用户功能，而不是只提交 helper、类型或状态处理。
- 同一 Phase 的 8 个任务都从同一个基线开始，不导入同 Phase 另一项尚未合并的实现。
- `pages/` 不做全局冻结。每份正式 Spec 会明确列出完成该功能预计需要修改的文件；如果确实需要修改 Page，就把对应 Page 写进该 Spec。
- 同一个 Page 在同一 Phase 只安排一位主要集成人。页面中的其他功能通过 Starter 已接好的 Component properties / events 独立实现，避免两个 PR 同时修改同一个 Page 文件。
- 文件列表用于划清并行边界，不是阻止合理修正。如果实现中发现必须修改列表之外的公共文件，应在 PR 中说明缺失的契约和修改理由，老师据此判断是当前 Spec 的必要工作还是 Starter 问题。
- 每项任务至少包含一组有业务意图的 Vitest，以及一条可在微信开发者工具中复现的用户旅程。Vitest 不检查 WXML 文本、固定颜色、Fixture 数量或微信框架内部行为。

## 2. 页面主要集成人

“主要集成人”表示该 Spec 可以根据自身用户旅程修改对应 Page 的控制器、WXML 或样式，并负责保留同页面其他已声明的组件入口。没有被分配为主要集成人的任务仍然可以交付完整的可见组件，只是不需要和同阶段同学争抢页面文件。

| Page | Phase 1 | Phase 2 |
| --- | --- | --- |
| Today | P1-01 Today 概览与主行动 | P2-01 近 7 天活动与今日记录 |
| 记录编辑 | P1-02 新建学习记录 | P2-02 编辑与删除单条记录 |
| 学习详情 | P1-04 学习记录详情 | 无新增页面级任务 |
| 学习日志 | P1-05 学习日志时间线 | P2-03 日期与标签筛选 |
| 学习统计 | P1-06 学习统计总览 | P2-05 本月学习日历 |
| 设置 | P1-07 学习偏好与关于 | P2-08 删除全部学习记录 |

P1-03、P1-08、P2-04、P2-06 和 P2-07 使用 Starter 已接好的页面入口完成各自的可见功能，不需要把 Page 作为自己的主要修改范围。

## 3. Phase 1：最小可用闭环

Phase 1 的共同起点是 `starter`。集成完成后，用户可以创建带主题的学习记录，在 Today、日志、统计和详情中看到结果，并通过微信身份把记录保存到云端。

| ID | 用户功能 | 难度 | 独立预览入口 |
| --- | --- | --- | --- |
| P1-01 | Today 概览与主行动 | 中 | Today + `empty`、`today`、`history` |
| P1-02 | 新建一条学习记录 | 中高 | 记录编辑（新建）+ `history` |
| P1-03 | 学习主题选择与新建标签 | 中 | 记录编辑（新建）+ `history` |
| P1-04 | 学习记录详情 | 低中 | 学习详情 + `history` 指定记录；另用 `read-error` |
| P1-05 | 学习日志时间线 | 中高 | 学习日志 + `empty`、`history`、`read-error` |
| P1-06 | 学习统计总览 | 高 | 学习统计 + `empty`、`history`、`read-error` |
| P1-07 | 学习偏好与关于 | 低中 | 设置 + `empty` |
| P1-08 | 云端记录与同步状态 | 高 | 设置 + Cloud 数据源模式 |

### P1-01 Today 概览与主行动

- 用户成果：用户能看见当前连续、今日分钟和今日记录数，并根据“首次使用”“有历史但今天未记录”“今天已有记录”得到不同的主行动文案；按钮可以进入新建记录。
- 主要文件：`features/today-summary/index.ts`、`components/today-summary/index.*`、`pages/today/index.*`、对应 Feature 测试。
- 独立验收：分别使用 `empty`、`history` 和 `today`；不依赖 P1-02 真正完成保存，只验收到达已注册的新建页。
- 测试意图：同日多条的分钟和数量、今天缺席时的连续口径、三种行动状态。
- 并行边界：不实现近 7 天日期条和今日记录列表，不修改 P2-01 的 Component / Feature。

### P1-02 新建一条学习记录

- 用户成果：用户可以填写学习内容、学习时长和今日收获，看到字段错误，处理未保存返回提醒，并在保存成功后回到来源页；保存中不能重复提交，失败后输入仍保留。
- 主要文件：`features/record-create/index.ts`、`components/record-editor/index.*`、`pages/record-edit/index.*`、对应 Feature 测试。
- 独立验收：从 Today 或日志进入新建页；使用 Local Repository 创建记录后返回并重新进入页面确认数据存在。
- 测试意图：内容清理和 1–300 字、时长 5–600 且步长为 5、收获最多 140 字、空收获不保存、无效输入不提交。
- 并行边界：保留 Starter 中的 `<tag-picker>` 接线；不实现标签内部交互，不实现编辑和删除。

### P1-03 学习主题选择与新建标签

- 用户成果：用户可以选择历史主题、输入新主题、移除已选主题，并始终看见当前完整选择；最多选择 3 个主题。
- 主要文件：`features/tag-picker/index.ts`、`components/tag-picker/index.*`、对应 Feature 测试。
- 独立验收：直接打开带 `history` Fixture 的新建页；即使 P1-02 仍是安全默认表单，标签区域也可以独立完成选择、新建、移除和错误反馈。
- 测试意图：从历史记录去重收集候选主题、去除首尾空白、拒绝空主题和超过 12 字的主题、最多 3 个、重复主题只保留一个。
- 并行边界：Starter 负责把历史候选主题和当前标签传给该组件，并把完整标签数组接回编辑器；P1-03 不导入 P1-02 的实现。

### P1-04 学习记录详情

- 用户成果：用户可以完整阅读日期、星期、创建时间、时长、内容、主题和今日收获；无主题或无收获时不出现空区块，并能进入编辑页。
- 主要文件：`features/record-detail/index.ts`、`components/record-detail/index.*`、`pages/record-detail/index.*`、对应 Feature 测试。
- 独立验收：编译模式直接用 `history` 打开固定记录；分别验证正常记录、不存在的 ID 和 `read-error`，不等待 P1-05 的日志入口。
- 测试意图：本地日期与时间展示、正文换行保留所需的视图模型、可选标签和收获的映射。
- 并行边界：详情页只提供编辑入口，不实现编辑或删除；不依赖 P1-05 的时间线。

### P1-05 学习日志时间线

- 用户成果：用户可以看到全部学习记录摘要和按日期倒序组织的时间线；同日记录按创建时间倒序，点击卡片可进入详情，空数据和读取失败有明确行动。
- 主要文件：`features/log-timeline/index.ts`、`components/log-timeline/index.*`、`pages/log/index.*`、对应 Feature 测试。
- 独立验收：使用 `history` 验证同日多条、跨日期和摘要；用 `empty` 与 `read-error` 验证空状态和重试。
- 测试意图：日期分组、组排序、组内排序、每组分钟、全部记录数、累计分钟和去重打卡日。
- 并行边界：Phase 1 只显示全部记录；保留结构化筛选和关键词搜索的安全插槽，不实现 P2-03 或 P2-04。

### P1-06 学习统计总览

- 用户成果：用户可以看到当前连续、最长连续、累计打卡日和累计学习分钟；没有记录时只看到引导行动，不出现空图表。
- 主要文件：`features/statistics-overview/index.ts`、`components/stats-overview/index.*`、`pages/stats/index.*`、对应 Feature 测试。
- 独立验收：使用 `history` 检查真实总览，使用 `empty` 检查首次状态，使用 `read-error` 检查重试。
- 测试意图：同日多条只算一个打卡日、分钟仍累加、今天断档但昨天有记录、跨月连续、空数据。
- 并行边界：复用 Starter 的连续日期函数；不实现日历、7 天趋势或主题排行，并保留三者入口。

### P1-07 学习偏好与关于

- 用户成果：用户可以读取并修改默认学习时长，看到保存中和失败状态；设置页同时展示版本和隐私说明入口。
- 主要文件：`features/preference/index.ts`、`components/settings-preference/index.*`、`components/settings-about/index.*`、`pages/settings/index.*`、对应 Feature 测试。
- 独立验收：把默认时长从 30 改为另一个合法值，退出并重新进入设置确认仍存在；再打开新建页确认使用新默认值。
- 测试意图：5–600、步长 5、非法值错误；已有学习记录不因偏好变化而改变。
- 并行边界：保留同步和危险区的组件入口；不修改 P1-08 或 P2-08 的 Component / Feature。

### P1-08 云端记录与同步状态

- 用户成果：学习记录通过当前微信身份保存和读取；设置页展示同步中、已同步和失败，并允许主动从云端重读。
- 主要文件：`repositories/cloud-record/`、Record Repository 的组合切换文件、`features/sync/index.ts`、`components/settings-sync/index.*`、对应 Repository / Feature 测试。
- 独立验收：使用 Cloud 数据源编译模式直接打开设置页验证连接和主动重读；通过 Repository 契约测试验证 list / get / create / update / remove，不依赖 P1-02 的表单先合并。
- 测试意图：当前用户查询条件、Cloud Document 双向映射、CRUD 生命周期、读取失败与不存在的区别、错误结果不泄露 `_openid` 或文档内容、同步状态变化。
- 并行边界：Starter 提供 CloudBase 初始化、环境 ID、collection、安全规则、数据源切换入口和 Document 映射骨架；P1-08 不创建云环境、不修改其他页面，也不实现删除全部记录。

## 4. Phase 1 集成闸门

8 个 P1 PR 合并后，老师按以下旅程验收共同基线：

1. 清空数据后打开三个 Tab，均显示正常空状态。
2. 从 Today 新建两条包含不同主题的今日记录。
3. Today、日志、统计和详情读取到同一批记录。
4. 重开小程序后记录仍从 CloudBase 读出。
5. 修改默认时长后新建记录，新记录采用新值，历史记录不变。
6. 制造读取失败，页面显示失败和重试，不把失败伪装成零记录。

只有这条旅程通过后，`main` 才作为 Phase 2 的共同起点。

## 5. Phase 2：回看与数据管理体验

Phase 2 的共同起点是 `phase-1-complete`。集成完成后，产品补齐 Today 活动、编辑删除、日志检索、统计洞察和全部数据删除。

| ID | 用户功能 | 难度 | 独立预览入口 |
| --- | --- | --- | --- |
| P2-01 | Today 近 7 天活动与今日记录 | 中 | Today + `today`、`history` |
| P2-02 | 编辑与删除单条记录 | 高 | 记录编辑（编辑）+ `history` 指定记录 |
| P2-03 | 日志日期与标签筛选 | 中高 | 学习日志 + `history`，另用外部 `date` / `tag` 参数 |
| P2-04 | 日志关键词检索与结果反馈 | 中 | 学习日志 + `history` |
| P2-05 | 本月学习日历 | 中高 | 学习统计 + `history` |
| P2-06 | 最近 7 天投入趋势 | 中高 | 学习统计 + `history` |
| P2-07 | 学习主题 Top 3 | 中 | 学习统计 + `history` |
| P2-08 | 删除我的全部学习记录 | 中高 | 设置 + Local `history`；Cloud 由当前用户验证 |

### P2-01 Today 近 7 天活动与今日记录

- 用户成果：用户能看到包含今天在内的 7 天活动标记和按创建时间倒序排列的今日记录；点击记录进入详情。
- 主要文件：`features/today-activity/index.ts`、`components/today-activity/index.*`、`pages/today/index.*`、对应 Feature 测试。
- 独立验收：使用 `today` 检查今日列表，使用 `history` 检查 7 天活动和历史记录不会混入今日列表。
- 测试意图：7 个本地日期、今日标记、无记录日期、跨月窗口、今日过滤与倒序。
- 并行边界：不改变 P1-01 概览口径；日期条只提供反馈，不承担日志筛选或跳转。

### P2-02 编辑与删除单条记录

- 用户成果：用户从详情进入编辑页后看到原值和原日期，可以修改可编辑字段；未保存离开需要确认；删除经过二次确认并返回原始 Tab。
- 主要文件：`features/record-edit-delete/index.ts`、`components/record-editor/index.*`、`pages/record-edit/index.*`、对应 Feature 测试。
- 独立验收：编译模式用 `history` 直接打开一条历史记录的编辑页，确认显示记录原日期；分别验证保存、放弃、删除取消、删除确认和写入失败。
- 测试意图：初始草稿与当前草稿的差异判断、编辑后不可变字段保持、`updatedAt` 更新、删除失败不产生假成功。
- 并行边界：复用 P1-02 的表单校验和 P1-03 的标签组件；不在详情页增加直接删除按钮。

### P2-03 日志日期与标签筛选

- 用户成果：用户可以选择或取消一个日期、选择一个主题或“全部”；日期和主题同时存在时按 AND 过滤，外部路由条件能正确恢复并清除。
- 主要文件：`features/log-structured-filter/index.ts`、`components/log-structured-filters/index.*`、`pages/log/index.*`、对应 Feature 测试。
- 独立验收：用 `history` 切换日期和主题；再通过统计路由形式直接带 `date` 或 `tag` 打开日志。
- 测试意图：7 天日期选项、历史主题选项、单条件和双条件 AND、超出 7 天的外部日期、取消与全部。
- 并行边界：保留关键词组件和组合入口；不实现关键词匹配，也不导入 P2-04。

### P2-04 日志关键词检索与结果反馈

- 用户成果：用户可以搜索学习内容和今日收获，实时看见结果条数与分钟数；无匹配时可以一键清除全部筛选。
- 主要文件：`features/log-keyword-filter/index.ts`、`components/log-keyword-search/index.*`、对应 Feature 测试。
- 独立验收：用 `history` 搜索只出现在内容、只出现在收获和完全不存在的关键词；纯逻辑测试使用一组预先筛选过的记录作为输入，证明关键词功能不会自行重置上游条件。
- 测试意图：首尾空白、内容匹配、收获匹配、无收获记录、结果条数与分钟、空关键词返回全部。
- 并行边界：页面负责按“结构化筛选后再关键词筛选”的顺序组合；P2-04 不导入或等待 P2-03。两个真实筛选条件的联合 UI 结果在 Phase 2 集成闸门验收。

### P2-05 本月学习日历

- 用户成果：用户看见当前本地月份的完整月历和有记录日期的标记，点击日期进入带日期条件的学习日志。
- 主要文件：`features/stats-calendar/index.ts`、`components/stats-calendar/index.*`、`pages/stats/index.*`、对应 Feature 测试。
- 独立验收：用 `history` 检查当前月排布、学习日标记和跳转；用 `empty` 确认没有伪造标记。
- 测试意图：月首星期偏移、月末天数、闰年二月、本月记录标记、跨月记录不误标。
- 并行边界：保留趋势和主题排行入口；不实现切换月份或年度热力图。

### P2-06 最近 7 天投入趋势

- 用户成果：用户能读出包含今天在内最近 7 个本地日期的每日学习分钟，没有记录的日期明确显示 0。
- 主要文件：`features/stats-seven-day-trend/index.ts`、`components/stats-seven-day-trend/index.*`、对应 Feature 测试。
- 独立验收：用 `history` 检查同日分钟累加、空缺日期和跨月日期；图形必须同时提供文字或数字。
- 测试意图：固定 7 项、日期顺序、同日累计、零值补齐、跨月与跨年。
- 并行边界：通过 Phase 1 已接好的统计页入口渲染；不修改 P2-05 日历或 P2-07 排行。

### P2-07 学习主题 Top 3

- 用户成果：用户看见使用次数最多的三个学习主题及次数，点击主题进入带主题条件的学习日志。
- 主要文件：`features/stats-tag-rank/index.ts`、`components/stats-tag-rank/index.*`、对应 Feature 测试。
- 独立验收：用 `history` 检查排序、只显示前三和带编码主题的跳转；无标签记录不进入排行。
- 测试意图：一条多标签记录分别计数、按记录数而非分钟、并列时按最近使用排序、少于三项和无标签。
- 并行边界：通过 Phase 1 已接好的统计页入口渲染；不修改日历或趋势。

### P2-08 删除我的全部学习记录

- 用户成果：设置页明确说明删除范围，用户经过二次确认后删除当前微信身份下的全部记录；失败时原数据保持，能够重试。
- 主要文件：`features/remove-all-records/index.ts`、`components/settings-danger-zone/index.*`、`pages/settings/index.*`、`repositories/cloud-record/` 中的批量删除实现、对应 Feature / Repository 测试。
- 独立验收：先用 Local `history` 验证取消、确认、成功和失败；Cloud 验收只操作当前学生自己的 `_openid` 数据，并由老师复核其他用户数据不受影响。
- 测试意图：二次确认状态、分批删除直到为空、任何一批失败都报告失败、查询始终限制当前用户、偏好不被删除。
- 并行边界：不删除微信身份、不实现账号注销、不清除默认时长；同 Phase 没有其他任务修改设置页或 Cloud Repository。

## 6. Phase 2 集成闸门

8 个 P2 PR 合并后，老师按以下旅程验收最终版本：

1. Today 显示 7 天活动和今日记录，点击记录能进入详情。
2. 编辑一条历史记录，日期和创建时间不变，内容与 `updatedAt` 更新。
3. 在日志组合日期、主题和关键词，列表与摘要同时变化，并能清除条件。
4. 从月历日期和主题排行进入日志，外部筛选条件正确生效。
5. 日历、7 天趋势和 Top 3 使用同一批真实记录得出一致结果。
6. 删除全部记录前可以取消；确认后当前用户所有记录消失，默认时长仍保留。
7. 任一读取或写入失败都显示失败，不出现无法恢复的假成功。
